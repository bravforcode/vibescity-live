## Critical Pitfalls

### Pitfall 1: PDPA-Violating Visitor ID Collection Without Consent Disclosure

**What goes wrong:**
The `vibe_visitor_id` (a UUID written to `localStorage` in `gamificationService.js`) qualifies as personal data under Thailand's PDPA. The PDPA defines personal data as "any information relating to a person which enables identification, whether directly or indirectly." A stable browser-scoped UUID that ties gamification history to a device session meets this definition. The current Privacy Policy (`PrivacyView.vue`) already discloses it, which is the correct instinct, but the app does not present a consent gate at first visit — data collection begins before the user has acknowledged the policy.

Since August 2025 Thailand's PDPC is actively imposing fines (eight cases in one public announcement), and enforcement moved from guidance to penalties. Offering coin rewards tied to the visitor_id adds a secondary PDPA risk: financial inducement to consent is considered potentially coercive, which can invalidate consent under PDPA principles.

**Why it happens:**
Developers treat a random UUID as "anonymous enough," not recognising that a persistent identifier enabling behavioural tracking is personal data regardless of whether a real name is attached.

**How to avoid:**
- Present a one-time consent banner (not a blocking modal) on first app load before `getVisitorId()` writes to `localStorage`.
- Consent must be: freely given, specific, informed, unambiguous, and logged (timestamp + version).
- The banner must explain: what is stored, why, and how to delete it (clear localStorage).
- Do not gate basic map-browsing behind consent — only gamification features require consent.
- Log consent timestamp + policy version in `localStorage` alongside the visitor_id.
- Provide a "Delete my data" control in SettingsPanel that clears the visitor_id and calls a Supabase Edge Function to purge associated gamification rows.

**Warning signs:**
- `getVisitorId()` is called before any user consent interaction.
- No `consent_given_at` or `consent_version` key exists in localStorage alongside `vibe_visitor_id`.
- Privacy Policy page exists but is only reachable through a settings drawer link.

**Phase to address:** The phase that wires the "Claim Your Vibe" flow (making gamification live). Must ship before the claim flow is activated.

### Pitfall 2: Gamification Farming via localStorage Reset (Visitor ID Cycling)

**What goes wrong:**
The entire anti-abuse model for daily check-ins and lucky wheel relies on the `vibe_visitor_id` stored in `localStorage`. Any user can:
1. Open DevTools → Application → LocalStorage → delete `vibe_visitor_id`.
2. Reload page. A new UUID is generated by `getVisitorId()`.
3. The Supabase RPCs (`claim_daily_checkin`, `spin_lucky_wheel`) treat this as a brand-new visitor.
4. Repeat indefinitely to farm coins.

Private browsing compounds this: iOS Safari private mode throws an exception on `localStorage.setItem`, so `getVisitorId()` falls into the catch block and generates a new UUID on every call (it cannot persist). Every session is a fresh visitor.

The `coinStore.js` also persists `coins`, `collectedVenues`, `achievements`, and `dailyStreak` to localStorage under key `vibe-coins`. Clearing this independently of the visitor_id desynchronises the client balance from the server balance.

**Why it happens:**
localStorage is developer-friendly and works without auth, but it provides zero tamper resistance. It is an appropriate session hint, not an abuse-prevention mechanism.

**How to avoid:**
- Server-side: implement IP-based rate limiting in the Supabase Edge Function or RLS policy. The existing `slowapi` limiter in the FastAPI backend cannot reach Supabase RPC calls made directly from the frontend.
- Server-side: add a `created_at` + `ip_hash` column to the gamification ledger table. A Postgres function can reject new visitor_ids that share an IP with more than N claims in 24h.
- Client-side: the visitor_id is necessary for UX continuity but must be treated as a hint, not a trust anchor. All reward caps must be enforced server-side.
- Consider Cloudflare Turnstile (free tier) on the claim action — it is invisible and does not require cookies, making it PDPA-compatible.
- Set a maximum lifetime reward per IP (e.g., 500 coins/day) enforced in the RPC, regardless of how many visitor_ids present.

**Warning signs:**
- The `claim_daily_checkin` RPC uses `p_visitor_id` as the sole dedup key with no IP cross-check.
- `coins` in `coinStore.js` is client-writable (no server reconciliation on page load).
- No Turnstile or CAPTCHA on any gamification action.

**Phase to address:** Same phase as the claim flow activation. Do not ship a live coin reward system without server-side caps.

### Pitfall 3: Anonymous Reviews With No Accountability Layer

**What goes wrong:**
`shopStore.addReview()` inserts directly into the `reviews` table via the Supabase client with `{ venue_id, ...review }`. There is no:
- Rate limit on the client path (the FastAPI `/ugc` routes have `@limiter.limit` but the frontend bypasses FastAPI entirely, going direct to Supabase).
- Content length validation before the insert.
- Duplicate/spam detection (same text submitted 10 times to inflate/destroy a venue rating).
- Moderation queue — reviews are live immediately.

With no login requirement, a competitor or bad actor can submit hundreds of 1-star reviews against a specific venue in seconds using automated requests to the Supabase PostgREST endpoint.

**Why it happens:**
Supabase's client library makes inserts easy. RLS policies are often set to `true` for authenticated users, but anonymous anon-key requests can also write if the table policy allows it. The `anon` key is public in the frontend bundle.

**How to avoid:**
- Move all review submissions through the FastAPI backend (not direct Supabase client) so `slowapi` rate limiting applies per-IP.
- Add RLS policy: `INSERT` on `reviews` allowed for `anon` role only if no row with same `visitor_id` + `venue_id` exists within the last 24 hours. Implement this as a Postgres row check, not client logic.
- Add a moderation status column (`status: 'pending' | 'approved' | 'rejected'`). New anonymous reviews default to `pending`. Display only `approved` reviews publicly.
- Content validation: max 1000 chars, minimum 10 chars, reject if content matches a known spam hash.
- Aggregate rating recalculation must be server-side (Postgres trigger) to prevent client-side rating manipulation.

**Warning signs:**
- `addReview()` calls `supabase.from("reviews").insert()` directly, not via FastAPI.
- No `status` column visible in the reviews select in `fetchShopReviews`.
- No rate limit error handling in the `addReview` action.

**Phase to address:** The phase that adds anonymous reviews. Must include a moderation queue and FastAPI proxy before going live.

### Pitfall 4: GPS Permission Denial Leaves Map Stuck on "Locating" State

**What goes wrong:**
`locationStore.startWatching()` calls `navigator.geolocation.watchPosition()`. On permission denial, `handlePositionError()` correctly sets `permissionStatus: 'denied'` and falls back to the Chiang Mai default. However, `isLoading` is reset to `false` in the error handler, but `isTracking` is set to `true` before `watchPosition()` is called and is never set to `false` on denial.

The deeper UX failure: there is no persistent UI message explaining why location is defaulted. Users who deny GPS see distance markers calculated from Chiang Mai center (18.7883, 98.9853) with no indication they are mock distances. Features that require real GPS (proximity-gated check-ins, "you are here" pin) silently fail or show incorrect data.

On iOS, the permission prompt appears once. If denied, `navigator.permissions.query({name:'geolocation'})` returns `'denied'` permanently until the user navigates to Settings > Safari > Location. There is no in-app way to re-trigger the prompt.

**Why it happens:**
Geolocation is treated as a binary on/off. The three distinct states (prompt/granted/denied) and two distinct error types (permission denied vs. position unavailable vs. timeout) are collapsed into a single fallback.

**How to avoid:**
- Differentiate `PERMISSION_DENIED` (permanent, user must go to Settings) from `POSITION_UNAVAILABLE` (transient, retry is reasonable) in the UI.
- On `PERMISSION_DENIED`: show a persistent non-blocking banner "Location access off — distances are approximate. Tap to learn how to enable."
- Gate proximity-dependent features (check-in within X meters) on `permissionStatus === 'granted'` AND `isMockLocation === false`. Show a locked state, not a silent failure.
- Do not call `startWatching()` silently on app init. Use a contextual trigger: "Show nearby venues" button requests location, with a one-sentence explanation before the OS prompt.
- Track `isTracking` correctly: set to `false` if permission is denied immediately in the error handler.

**Warning signs:**
- `startWatching()` is called automatically in `initializeStores()` without user action.
- `isMockLocation` is set but there is no component that checks it before rendering distance-dependent UI.
- The `permissionStatus: 'denied'` state has no distinct UI treatment in any component.

**Phase to address:** The phase that activates proximity-gated check-ins. The existing GPS code is functional for display; the UX failure emerges when check-ins require real proximity.

## Moderate Pitfalls

### Pitfall 5: Pinia Persist Plugin Archived — Upgrade Risk on Next Install

**What goes wrong:**
`pinia-plugin-persistedstate` v4.7.1 is in use. The GitHub repository was archived by its author on August 31, 2025. The project has migrated to Codeberg (`codeberg.org/praz/pinia-plugin-persistedstate`). The npm package still installs and works, but the GitHub repo will receive no security patches, and npm installs of future Vue/Pinia major versions may pull incompatible peer deps without a maintained resolution path.

The immediate practical risk: six stores (`coinStore`, `locationStore`, `shopStore`, `userPreferencesStore`, `favoritesStore`, `userStore`) use the `persist` option. If the plugin hydrates stale data from a previous schema version, computed properties that depend on the shape of persisted arrays (e.g., `collectedVenues` expected as `Array<venueId>`, `achievements` expected as `Array<string>`) will silently misbehave if the shape changes between deployments.

**How to avoid:**
- Pin the exact version in `package.json` (already done with `^4.7.1` — change to `4.7.1` to prevent patch drift).
- Add a schema version key to each persisted store. On hydration, if `schemaVersion` mismatches the expected version, call `$reset()` and re-fetch from server.
- For `coinStore`: after hydration, always reconcile `coins` balance from server on first authenticated load (`fetchUserStats` already does this for authenticated users; extend it to visitor IDs via the gamification RPCs).
- Monitor Codeberg releases for security disclosures. The Codeberg fork is the maintained version.

**Warning signs:**
- `collectedVenues` is persisted as an array but consumed as a Set (`collectedSet = new Set(collectedVenues.value)`). If an old localStorage value contains a non-array (e.g., from a previous store version), `new Set()` will not throw but will produce an empty Set silently.
- No schema version check exists in any store's hydration path.

**Phase to address:** Any phase that changes the shape of persisted state. Add migration guard before deploying schema-breaking changes.

### Pitfall 6: Lucky Wheel `box-shadow` Causes Composite Layer Explosion on Mid-Range Android

**What goes wrong:**
`LuckyWheel.vue` applies:
- `box-shadow: 0 0 30px rgba(139, 92, 246, 0.5)` on `.wheel` (static, not animated — OK).
- `box-shadow: 0 4px 20px rgba(139, 92, 246, 0.5)` on `.wheel-center` (static — OK).
- `transition: transform 4s cubic-bezier(0.17, 0.67, 0.12, 0.99)` on `.wheel` during spin.

The spin itself animates `transform: rotate()` on the `.wheel` div. This is GPU-composited and should be performant. However, the existing CLAUDE.md constraint "no animated box-shadow (CPU bug known)" was triggered in earlier phases. The `.wheel` has a static box-shadow on the same element being transform-animated. On mid-range Android (Snapdragon 680 class, which is common in Thailand's ฿6,000–฿10,000 phone market), animating `transform` on an element with `box-shadow` can force the browser to repaint the shadow on each frame instead of promoting it to its own layer, causing frame drops.

Additionally, `backdrop-filter: blur(8px)` on `.wheel-overlay` is active the entire time the wheel is spinning. This is one of the most expensive CSS properties on mobile. Simultaneously rendering backdrop blur + rotating transform + glow shadows can drop below 30fps on low-tier devices.

**How to avoid:**
- Isolate the spinning element from the shadow: wrap `.wheel` in an outer `.wheel-glow` div that holds the `box-shadow`. The inner `.wheel` div has only `transform` and no shadow. This forces the browser to promote them to separate layers.
- Replace `backdrop-filter: blur(8px)` on the overlay with `background: rgba(0,0,0,0.9)` (no blur). The existing overlay already uses `rgba(0,0,0,0.9)` — the backdrop-filter is additive on top.
- Gate the `4s cubic-bezier` spin transition behind `prefersReducedMotion` check (already done for the 120ms fallback) AND a `motionBudget` check from `userPreferencesStore`. If `motionBudget === 'micro'`, use the 120ms transition unconditionally.
- Add `will-change: transform` to `.wheel` while spinning only (add/remove via `:class` binding on `isSpinning`). Remove it after spin completes to free the layer.

**Warning signs:**
- Jank reported on "colorful" or "cinematic" map preset (which enables more visual effects simultaneously with the wheel).
- Chrome DevTools Performance shows "Recalculate Style" or "Paint" frames during the spin animation.
- The wheel overlay is opened while Mapbox is rendering (both backdrop-filter and Mapbox WebGL active simultaneously).

**Phase to address:** The phase that makes the Lucky Wheel interactive (not just UI stub). Test on a mid-range Android before shipping.

## Minor Pitfalls

### Pitfall 7: Visitor ID Private Browsing Reset Breaks Gamification UX

**What goes wrong:**
In iOS Safari private mode, `localStorage.setItem()` throws `QuotaExceededError`. The `getVisitorId()` try/catch correctly catches this and returns a new `crypto.randomUUID()` — but does not persist it. Every call to `getVisitorId()` in a private session generates a different UUID, so the RPC server sees a new visitor on every gamification action within the same session. The daily check-in will appear claimable on every page reload.

This is a UX issue (confusing state) and an abuse surface (if the server does not rate-limit by IP).

**How to avoid:**
- Fall through to `sessionStorage` if `localStorage` throws. `sessionStorage` works in iOS Safari private mode and persists the visitor_id for the tab session.
- Update `getVisitorId()` with a two-level storage fallback: try `localStorage`, then `sessionStorage`, then return an in-memory singleton (module-level variable).
- Communicate clearly in the UI: "Your rewards are saved to this device. Use private/incognito mode? Rewards won't be saved."

**Phase to address:** Same phase as visitor ID persistence design. Low effort fix, moderate UX impact.

### Pitfall 8: `addReview` Persists Reviews to localStorage via shopStore

**What goes wrong:**
`shopStore` persists the `reviews` key: `persist: { paths: ["reviews"], key: "vibe-shops" }`. The `reviews` ref is an object keyed by `shopId`, each holding an array of review objects. If a user views 50 venues and reads reviews for each, this object can grow unboundedly in localStorage. The 5MB localStorage quota is shared across all keys. When combined with the other six stores persisting data, review cache bloat can cause `QuotaExceededError` for the preference store, breaking the motion/haptics settings on next load.

**How to avoid:**
- Cap the reviews cache: store only the 5 most recent reviews per venue, not the full 50-review array returned by `fetchShopReviews`.
- Add an LRU eviction strategy: keep only reviews for the last 10 viewed venues.
- Alternatively, remove `reviews` from the persist paths entirely — reviews are cheap to re-fetch and should not consume persistent quota.

**Phase to address:** Before the anonymous reviews feature ships (adding more review volume).

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Client-side achievement checks in `coinStore.checkForAchievements()` | Fast UI feedback, no server round-trip | Server cannot verify; achievements can be self-granted by editing localStorage | MVP only — replace with server-authoritative achievement grants before any achievement has real redemption value |
| `spendCoins` mutates `coins` locally before server write | Instant UI | If server write fails silently (`.catch(console.error)` swallows the error), balance desync is permanent | Never for any coin action with real-world value (e.g., redeemable prizes) |
| Anonymous reviews written directly via Supabase `anon` key | Simple, no backend needed | Bypasses all FastAPI rate limiting; full write access if RLS is misconfigured | Never in production with open anonymous write |
| `vibe_visitor_id` as sole gamification identity | No login friction | Trivially bypassed by clearing localStorage; incompatible with cross-device continuity | Acceptable for casual engagement, never for redeemable rewards |
| `pinia-plugin-persistedstate` without schema versioning | Data survives reload | Silent misbehaviour after store shape changes | Never when persisted shape changes between deploys |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Supabase RPC from frontend | Treating RPC as server-authoritative because it runs in Postgres — but the `anon` key allows calling it without authentication | All gamification RPCs must check `p_visitor_id` AND an IP-hash rate limit inside the Postgres function, not just client-side guards |
| Supabase direct client `insert` (reviews) | Bypasses FastAPI rate limiting entirely | Route all writes through FastAPI so `slowapi` applies; or add a custom rate-limit Postgres function |
| `pinia-plugin-persistedstate` hydration | State is hydrated synchronously before the app mounts, but some state (e.g., `coins`) is overwritten by an async `fetchUserStats`. The window between hydration and server reconciliation can cause a coin balance flash | Use `isHydrated` flag; suppress balance display until server data confirms |
| iOS GPS permission re-request | `getCurrentPosition()` silently uses the cached `denied` result with no new OS prompt | Detect `permissionStatus === 'denied'`, show a Settings deep-link, do not call `getCurrentPosition()` again |
| LuckyWheel + Mapbox simultaneously active | Mapbox WebGL + backdrop-filter + wheel spin animation = three GPU surfaces active at once on mobile | Pause Mapbox render or close map layer while wheel overlay is open |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| `backdrop-filter: blur` on modal overlay while Mapbox renders | Dropped frames during wheel spin on mid-range Android | Remove backdrop-filter from wheel/checkin overlays; use opaque backgrounds | Any device below Snapdragon 730 equivalent |
| Unbounded `reviews` object in persisted shopStore | `QuotaExceededError` on localStorage write for any subsequent store | Cap to 5 reviews per venue, LRU evict to 10 venues | After ~50 venue views with review data |
| `collectedVenues` array grows with every check-in, persisted indefinitely | localStorage reads slow as array grows; computed `collectedSet` rebuilt on every reactivity cycle | Store only venue IDs (already doing this), but add a max of 500; archive older entries to server-side profile | At ~500 check-ins (power users) |
| `transform: rotate` on element with `box-shadow` | Jank during wheel spin on low-end devices | Separate shadow to a parent layer, add `will-change: transform` during spin only | Mid-range Android with concurrent visual effects |
| `gamificationService` awaits `supabase.auth.getUser()` on every gamification call | 150–300ms added latency before every claim/spin/check-in | Cache auth state in a module-level variable with a short TTL (e.g., 5s); only re-fetch on auth state change events | Every time a gamification action is triggered |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Reviews inserted via anon Supabase key with no RLS write guard | Any automated script can flood any venue with fake reviews using the public `anon` key | Add RLS INSERT policy requiring a visitor_id + rate-check; route through FastAPI for IP limiting |
| `spendCoins` error swallowed with `.catch(console.error)` | Silent balance desync — user loses coins with no record | Log to a dead-letter table; at minimum, throw and surface an error state to the user |
| No consent log for `vibe_visitor_id` creation | PDPA violation — collecting personal data without documented consent | Write `consent_given_at` + `consent_policy_version` to localStorage alongside visitor_id before first gamification action |
| Visitor ID used as trust anchor for coin grants | Coin farming via localStorage clear, private browsing | IP-based cap inside Supabase RPC; Cloudflare Turnstile on claim actions |
| `userPreferencesStore` persists home location GPS coordinates | Exact home address stored in cleartext localStorage | Acceptable for UX; document in Privacy Policy as "precise location stored locally, never transmitted without consent" — this disclosure is currently missing from PrivacyView.vue |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| GPS permission denied → silent mock location with no indicator | User sees "500m away" for venues actually 2km away; check-in gating confuses users | Show a persistent low-priority banner: "Using Chiang Mai center — tap to enable location" |
| Geolocation prompt on app load (before user action) | iOS/Android users who deny the prompt can never re-enable it in-session | Trigger location request from a user action ("Find places near me") with a one-line explanation beforehand |
| Lucky wheel closed then re-opened resets `rotation` to 0 | Wheel jumps to 0deg visually, breaking the "already spun today" visual continuity | Persist `rotation` value for the session (sessionStorage or component-level keep-alive) |
| Anonymous reviewer has no edit/delete path | User submits wrong review, cannot correct it | Show "Your review" indicator tied to visitor_id with an edit window of 15 minutes after submission |
| Check-in claim from VibeActionSheet is currently a no-op stub | "Claim Your Vibe" button does nothing silently | Show a disabled state with tooltip "Coming soon" until the claim flow is wired — no silent no-ops on primary CTA |

## "Looks Done But Isn't" Checklist

- [ ] **Claim flow:** `VibeActionSheet.vue` emits `"claim"` but the parent handler in `HomeView.vue` is a stub. Verify the handler calls `coinStore.checkIn()` with real GPS proximity validation before marking complete.
- [ ] **Consent gate:** `getVisitorId()` writes to localStorage. Verify a consent acknowledgement is shown and recorded before this write happens on first visit.
- [ ] **Server-side caps:** `claim_daily_checkin` and `spin_lucky_wheel` Supabase RPCs. Verify they enforce IP-rate-limiting inside the Postgres function, not just client-side `canClaimToday / canSpinToday` guards.
- [ ] **Review RLS:** `reviews` table in Supabase. Verify the INSERT policy does not allow unbounded anonymous writes with the `anon` key.
- [ ] **GPS denial state:** Verify `permissionStatus === 'denied'` triggers a visible UI indicator and does not silently present mock distances as real.
- [ ] **Lucky wheel shadow isolation:** Verify `.wheel` does not have `box-shadow` on the same element as `transform: rotate` animation. Run Chrome DevTools "Rendering > Paint Flashing" during a spin.
- [ ] **Persist schema version:** Verify `coinStore` does not silently misbehave when `collectedVenues` is an older shape. Add a `_schemaVersion` check.
- [ ] **Private browsing fallback:** Verify `getVisitorId()` does not generate a new UUID on every call in iOS Safari private mode. Check by wrapping localStorage in a sessionStorage fallback.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| PDPA consent not collected before launch | HIGH | Retroactive: add consent banner, notify users via in-app message, create deletion mechanism, file with PDPC if required |
| Coin farming discovered post-launch | MEDIUM | Add IP-rate-limit to RPCs (DB migration, no deploy needed), reset suspicious balances, add Turnstile |
| Review spam flood damages venue ratings | MEDIUM | Add `status: 'pending'` column (DB migration), queue existing reviews for moderation, add RLS policy |
| Pinia persist schema mismatch after deploy | LOW | Deploy schema version check + `$reset()` guard; affected users get a fresh store, lose local preferences |
| GPS denial causing incorrect proximity check-ins | LOW | Add `isMockLocation` guard to check-in gate; retroactive: void check-ins where `isMockLocation` was true |
| Lucky wheel jank on low-end Android | LOW | Fix CSS layer isolation (no deploy needed, style change only), ship as hotfix |
